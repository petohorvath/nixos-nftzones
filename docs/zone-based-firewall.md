# Zone-Based Firewall

A zone-based firewall is an abstraction layer over a packet filter (here, nftables). Interfaces and addresses are first grouped into named **zones**; rules and policies are then expressed between zones rather than against raw `iifname` / `ip saddr` matches. The zone layer compiles to ordinary nftables rules at the bottom.

## Why Zones

Without zones, every rule restates topology:

```
iifname "eth1" ip saddr 10.0.0.0/24 oifname "eth0" tcp dport 443 accept
iifname "eth2" ip saddr 10.0.1.0/24 oifname "eth0" tcp dport 443 accept
iifname "eth1" ip saddr 10.0.0.0/24 oifname "wg0"  tcp dport 443 accept
...
```

Every interface and subnet appears in dozens of rules. Adding a new LAN segment means editing each rule. Zones replace those repeated matches with a name:

```
zone lan { interfaces = eth1, eth2; networks = 10.0.0.0/24, 10.0.1.0/24 }
zone wan { interfaces = eth0 }
zone vpn { interfaces = wg0 }

rule lan -> wan,vpn tcp dport 443 accept
```

Adding a third LAN interface is now a one-line change to the `lan` zone.

## What a Zone Is

A zone is a named grouping of one or more:

- **Interfaces** — by name, e.g. `eth0`, `wg0`, or by wildcard, e.g. `wlan*`.
- **Networks** — IPv4 or IPv6 prefixes, e.g. `10.0.0.0/24`, `2001:db8::/32`.
- **Addresses** — single hosts, e.g. `10.0.0.5`.

A zone may mix kinds. Membership of a packet is determined per packet by checking its ingress (or egress) interface and/or its source (or destination) address against the zone's members.

| Member kind | Matched against (source side) | Matched against (destination side) |
|---|---|---|
| interface | `iif` / `iifname` | `oif` / `oifname` |
| network or address | `ip saddr` / `ip6 saddr` | `ip daddr` / `ip6 daddr` |

A zone with only interfaces is purely topological. A zone with only networks is purely address-based and ignores which interface a packet came in on. Mixed zones combine both predicates (typically with logical AND, so a packet must satisfy both to belong).

## Source and Destination Zones

Every packet seen by the firewall has two zone identities:

- **Source zone** — derived from the ingress interface and source address.
- **Destination zone** — derived from the egress interface and destination address.

Rules and policies are directed: they target an ordered pair `src → dst`. Allowing traffic in one direction does not implicitly allow it in the other (return traffic is handled separately, see *Stateful Traffic*).

```
              src zone           dst zone
   packet ─▶  (where it came)    (where it is going)
                          │
                          └── rule lookup: src → dst
```

## Policies

A **policy** is the default verdict for an entire `src → dst` zone pair. It is consulted when no specific rule matches.

```
policy lan -> wan accept     # LAN out to internet: allowed by default
policy wan -> lan drop       # internet in to LAN: blocked by default
policy lan -> local accept   # LAN can reach the firewall itself
policy all -> all drop       # implicit catch-all: deny
```

Common defaults:

| Direction | Typical policy |
|---|---|
| trusted → untrusted | `accept` |
| untrusted → trusted | `drop` |
| trusted → local | `accept` |
| untrusted → local | `drop` |
| local → all | `accept` |

## Rules

A **rule** is a per-zone-pair exception that refines or contradicts the policy. Rules use the same matching syntax as nftables (ports, protocols, sets, conntrack states, etc.) but are scoped to a directed zone pair.

```
rule wan -> local tcp dport 22 accept comment "ssh from anywhere"
rule wan -> local tcp dport { 80, 443 } accept comment "public web"
rule lan -> wan udp dport 53 accept                          # DNS out
rule guest -> lan drop                                       # isolate guests
```

Rule order within a zone pair follows the same first-match semantics as nftables. A rule that issues `accept` or `drop` ends evaluation for that pair; the policy applies only when no rule matches.

## Intra-Zone Traffic

A zone may have rules to itself. This is the natural place for "members of this zone may (or may not) talk to each other":

```
policy lan -> lan accept     # LAN hosts can reach each other
policy guest -> guest drop   # guests are isolated from each other
```

Without an intra-zone policy, the implementation's catch-all default applies (typically `drop`).

## Special Zones

A zone-based firewall recognises two configurable special zone names — both implementations use them, even if they pick different defaults:

| Special zone | Default name | Meaning |
|---|---|---|
| local zone | `local` | The firewall machine itself. Configurable via `settings.localZone`. |
| wildcard zone | `all` | Matches every declared zone. Used for broad policies and catch-alls. Configurable via `settings.wildcardZone`. |

The local zone is interpreted by direction:

| Position | Meaning |
|---|---|
| local as **source** | Packet was generated by a local process. Matches in the `output` hook. |
| local as **destination** | Packet terminates on the firewall. Matches in the `input` hook. |
| local on neither side | Forwarded packet. Matches in the `forward` hook. |

The wildcard zone is mostly used in policies (`policy all -> all drop`) and broad allow rules (`rule local -> all accept`).

## Zones beyond interfaces and CIDRs

The standard zone model expresses membership as "this set of interfaces plus this set of CIDRs". For zones whose membership is defined differently — fwmark-tagged traffic, VLAN-id-bound flows, cgroup-classified packets — the `matchOverride` field provides per-direction, per-section overrides:

```nix
zones.vpn-users = {
  matchOverride.ingress.extra = [ (eq meta.mark 0x100) ];
  matchOverride.egress.extra  = [ (eq meta.mark 0x100) ];
};
```

Each side has four nullable sections:

| Section      | Substitutes for                                          |
|--------------|----------------------------------------------------------|
| `interfaces` | the auto `<ifField> @<zone>_iifs` interface clause       |
| `ipv4`       | the auto `ip <addr> @<zone>_v4` family clause            |
| `ipv6`       | the auto `ip6 <addr> @<zone>_v6` family clause           |
| `extra`      | family-agnostic prefix (mark, vlan, cgroup, …); no auto  |

Override sections compose with the auto path section-by-section. Setting `ipv4` only overrides v4; v6 still uses the auto set if the zone has v6 CIDRs. Setting `extra` adds clauses to every variant without disabling the auto path.

The `interfaces` section is hook-gated — it's only emitted at hooks where the relevant `iifname` / `oifname` field is valid (e.g. dropped at `output` for from-direction). The other sections work at any hook.

## Stateful Traffic

Zones describe direction-keyed *policy*; on their own they say nothing about how reply packets are handled. Two designs are possible:

- **Stateless.** Every packet is classified into a zone pair independently. A reply from `wan` to `lan` requires an explicit `wan -> lan` rule, which forces most useful policies to be written symmetrically in both directions and makes asymmetric defaults like `lan -> wan accept` / `wan -> lan drop` unusable for any real connection.
- **Stateful (conntrack).** Connections are tracked, and packets matching `ct state established,related` are accepted up front, before any zone-pair rule runs. A `lan -> wan accept` policy then implicitly allows return traffic, because the reply matches the established state and never reaches the `wan -> lan` policy.

Conntrack is not strictly required by the zone model — it is the practical layer that makes asymmetric zone policies useful. nftables provides it via the `ct` expression and the kernel's conntrack subsystem; this library assumes the stateful model and emits `ct state established,related accept` early in each base chain.

Consequences:

- A unidirectional `policy a -> b accept` permits return traffic from `b` to `a` automatically. No reverse rule is needed for replies.
- New connections in the reverse direction are still subject to `policy b -> a`.
- Invalid packets (`ct state invalid`) are dropped early, before zone classification, since they correspond to no legitimate state.

## Mapping to nftables

The zone layer compiles to a single inet table. Sketch:

```
table inet zonefw {
    # interface membership
    set lan_iifs   { type ifname; elements = { eth1, eth2 } }
    set wan_iifs   { type ifname; elements = { eth0 } }

    # network membership
    set lan_nets   { type ipv4_addr; flags interval; elements = { 10.0.0.0/24, 10.0.1.0/24 } }

    chain forward {
        type filter hook forward priority filter
        policy drop

        ct state established,related accept
        ct state invalid drop

        # rule: lan -> wan tcp dport 443 accept
        iifname @lan_iifs ip saddr @lan_nets oifname @wan_iifs tcp dport 443 accept
    }

    chain input {
        type filter hook input priority filter
        policy drop

        iif lo accept
        ct state established,related accept

        # rule: lan -> local accept
        iifname @lan_iifs ip saddr @lan_nets accept
    }

    chain output {
        type filter hook output priority filter
        policy accept   # rule: local -> all accept
    }
}
```

Key compilation choices:

- **Sets for membership.** Interfaces and networks become named sets; zone matches become set lookups. Adding a member to a zone is an in-place set update, not a ruleset rebuild.
- **Hook selection by direction.** The local zone as destination routes rules into `input`; as source into `output`; everything else into `forward`.
- **Conntrack short-circuit.** `ct state established,related accept` runs before any zone-pair rule, providing automatic return traffic.
- **Default drop.** Base chains use `policy drop`; an explicit zone-pair `accept` is required for traffic to pass.

The zone language stays declarative; the mapping to chains, hooks, and sets is a generation detail.
